# Section 4.5 Database Design (ERD)

The Senyra Learning Platform uses a relational database design implemented locally in SQLite and accessed through a FastAPI backend using SQLAlchemy. The database supports the main roles of the system, including students, teachers, and administrators, and stores information required for authentication, role-based access control, learning content, assessments, progress tracking, analytics, recommendations, feedback, interventions, audit logging, and dashboards.

The inspected database contains 28 application tables. The central entity is the `students` table, which stores user account records and includes a `role` attribute to distinguish students, teachers, and administrators. This account table is referenced by most learner-facing and teacher-facing tables, including quiz attempts, learning events, class groups, notifications, feedback messages, intervention plans, audit logs, Learning DNA profiles, creativity assessments, and flow-state records. Although the table is named `students`, it functions as the platform's general user table.

The learning content area of the schema is organised around `courses`, `lessons`, `quizzes`, and `questions`. A course can have multiple lessons and multiple quizzes. Each quiz can contain multiple questions, and student performance is stored separately in `quiz_attempts`. This separation supports normalization because course content, assessment questions, and learner attempts are stored as distinct entities rather than being duplicated in one table.

Student progress is captured through several related tables. `lesson_progress` records completed lessons and uses a unique constraint on `student_id` and `lesson_id` to prevent duplicate completion records. `quiz_attempts` stores quiz scores over time. `subject_quiz_results` stores subject and topic-level performance, while `weak_topics` stores detected areas where a learner may need additional support. These tables provide the foundation for personalised recommendations, weak topic detection, teacher dashboards, and intervention planning.

The database also supports advanced analytics and intelligent learning features. `learning_events` records generic learning activity events, which can be used as input for dashboards and machine learning models. `recommendation_history` records the recommendations generated for a learner and whether the learner found them useful. `learning_dna_profiles` stores each learner's calculated Learning DNA profile, while `learning_dna_questionnaire_responses` stores the raw questionnaire responses used to explain or derive the profile. `flow_sessions` and `flow_summaries` support flow state detection by storing engagement, duration, performance, and aggregate flow metrics. `creativity_assessments` and `creativity_responses` support the creativity engine by storing both summary scores and detailed student responses.

Teacher and administration features are represented through `class_groups`, `class_enrollments`, `feedback_messages`, `student_notes`, `intervention_plans`, `notifications`, and `audit_logs`. Class groups are linked to teachers, while enrollments link students to those groups. Feedback, notes, and intervention plans allow teachers to document support actions. Audit logs provide accountability by recording important user actions, which is important for administrative monitoring and system transparency.

SQLite was selected for the local implementation because it is lightweight, file-based, and easy to deploy during development. It does not require a separate database server, which makes it suitable for a final-year project prototype and demonstration environment. SQLite also works well with DB Browser for SQLite, allowing the schema, tables, and relationships to be inspected visually for project documentation.

The design is also prepared for future migration to PostgreSQL. The backend uses SQLAlchemy and a configurable `DATABASE_URL`, which means the database engine can be changed without rewriting the application data-access layer. The schema uses common relational types such as integer primary keys, text fields, booleans, floating-point scores, and datetime fields, all of which have PostgreSQL equivalents. A future production version should add Alembic migrations, stronger constraints for roles and score ranges, additional indexes for analytics queries, and more explicit foreign keys for subject and topic references.

Overall, the database design supports both transactional learning-platform features and analytical features. Transactional tables store users, courses, lessons, quizzes, feedback, and interventions. Event and history tables store behavioural records over time, enabling dashboards, machine learning features, cognitive risk prediction, explainable AI outputs, and personalised recommendations. This combination makes the schema appropriate for an adaptive learning platform because it records both the educational content structure and the learner behaviour needed for intelligent decision-making.
